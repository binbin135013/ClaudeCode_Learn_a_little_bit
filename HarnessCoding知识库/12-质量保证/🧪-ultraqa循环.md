# 🧪 UltraQA 循环

## 1. 概述

UltraQA 是 Claude Code 中的质量保证核心工作流，通过**测试-验证-修复-重复**的循环确保代码质量。

```
┌─────────────────────────────────────────────────────────┐
│                      UltraQA 循环                        │
│                                                         │
│    ┌─────────┐    ┌─────────┐    ┌─────────┐            │
│    │  测试   │ ──▶│  验证   │ ──▶│  修复   │            │
│    └─────────┘    └─────────┘    └─────────┘            │
│         ▲                                        │      │
│         └────────────────────────────────────────┘      │
│                         重复                             │
└─────────────────────────────────────────────────────────┘
```

## 2. UltraQA 工作流详解

### 2.1 测试阶段 (Test)

**目标**：编写可执行的质量规范

| 活动 | 产出 | 工具/命令 |
|-----|------|---------|
| 编写单元测试 | `*.test.ts` | Bash/Edit |
| 编写集成测试 | `*.integration.test.ts` | Bash |
| 编写端到端测试 | `*.e2e.test.ts` | Bash |
| 定义验收标准 | 文档化的检查项 | Write |

**关键原则**：
- 测试必须**先于**实现编写（符合TDD）
- 每个功能至少有一个对应测试
- 测试用例命名必须清晰描述预期行为

### 2.2 验证阶段 (Verify)

**目标**：确认测试通过且符合质量标准

| 验证类型 | 检查内容 | 通过标准 |
|---------|---------|---------|
| 语法检查 | TypeScript编译 | 零错误 |
| 单元测试 | 覆盖率、通过率 | ≥80%覆盖率 |
| 类型检查 | TSC --noEmit | 零错误 |
| Lint检查 | ESLint规则 | 零警告 |

**验证命令模板**：
```bash
# 顺序执行质量门禁
npm run type-check && npm run lint && npm run test
```

### 2.3 修复阶段 (Fix)

**目标**：消除验证阶段发现的所有问题

**优先级处理**：
1. **P0 - 阻断**：编译错误、测试失败 → 必须立即修复
2. **P1 - 高**：Lint警告、类型警告 → 修复后再继续
3. **P2 - 中**：覆盖率不足 → 补充测试
4. **P3 - 低**：代码风格问题 → 优化但不阻断

**修复流程**：
```
发现问题 → 分析根因 → 应用修复 → 重新验证 → 通过则继续
```

### 2.4 重复阶段 (Repeat)

**目标**：持续迭代直到满足所有质量标准

**退出条件**：
- 所有测试通过
- 覆盖率 ≥ 阈值
- 零阻断问题
- 代码审查通过

## 3. TDD 模式在 Claude Code 中的应用

### 3.1 Red-Green-Refactor 循环

```
┌──────────────────────────────────────────────────────┐
│  Red          Green              Refactor            │
│   │             │                  │                │
│   ▼             ▼                  ▼                │
│ 写一个   →   写最简单   →   重构代码                  │
│ 失败的测试   通过的实现   保持功能不变                │
└──────────────────────────────────────────────────────┘
```

### 3.2 Claude Code 中的 TDD 工作流

**步骤1 - 写测试（Red）**：
```
用户：实现一个计算阶乘的函数
Claude：
1. 先创建测试文件 factorial.test.ts
2. 编写测试用例覆盖各种输入
3. 运行测试，确认失败
```

**步骤2 - 写实现（Green）**：
```
Claude：根据测试要求，写最简单的实现让测试通过
- 不追求完美，只求通过
- 后续再优化
```

**步骤3 - 重构（Refactor）**：
```
Claude：在保持功能的前提下优化代码结构
- 提取重复逻辑
- 改善命名
- 提升性能
```

### 3.3 TDD 命令模式

| 场景 | 命令 | 说明 |
|-----|------|-----|
| 启动TDD模式 | `/tdd` | 激活TDD技能 |
| 写测试优先 | `写测试，然后实现` | 明确指示测试先行 |
| 验证测试覆盖 | `检查测试覆盖率` | 查看覆盖报告 |

## 4. 测试覆盖率阈值与质量门

### 4.1 覆盖率等级

| 等级 | 语句覆盖 | 分支覆盖 | 路径覆盖 | 适用场景 |
|-----|---------|---------|---------|---------|
| **基础** | ≥70% | ≥60% | ≥50% | 原型、实验项目 |
| **标准** | ≥80% | ≥70% | ≥60% | 生产项目 |
| **严格** | ≥90% | ≥85% | ≥75% | 关键业务系统 |

### 4.2 质量门标准

```yaml
质量门配置:
  blocking_issues:
    - compilation_error
    - test_failure
    - type_error

  non_blocking_issues:
    - lint_warning
    - low_coverage

  thresholds:
    statement_coverage: 80
    branch_coverage: 70
    function_coverage: 90
```

### 4.3 覆盖率报告生成

```bash
# 生成覆盖率报告
npm run test:coverage

# 查看HTML报告
open coverage/lcov-report/index.html
```

## 5. UltraQA 与 everything-claude-code:tdd-workflow

### 5.1 整合要点

| tdd-workflow要素 | UltraQA对应 | 整合方式 |
|-----------------|-------------|---------|
| 快速反馈循环 | 验证阶段 | 每次修改后自动验证 |
| 测试即文档 | 测试用例 | 测试名描述行为 |
| 小步前进 | 重复阶段 | 每次迭代完成一个小功能 |
| 安全网 | 覆盖率阈值 | 回归保护 |

### 5.2 实践检查清单

- [ ] 每个功能都有对应测试
- [ ] 先写测试再写实现（Red→Green）
- [ ] 测试失败时不写新代码
- [ ] 重构时保持测试通过
- [ ] 覆盖率达标后再提交
- [ ] 代码审查覆盖测试质量

## 6. 常见问题与解决

| 问题 | 原因 | 解决方案 |
|-----|------|---------|
| 测试运行慢 | 异步操作未mock | 使用 fake/timer |
| 覆盖率不足 | 边界条件未覆盖 | 补充边界测试用例 |
| Flaky测试 | 依赖顺序/时间 | 隔离测试、独立运行 |
| 难以测试 | 耦合严重 | 重构为可测试结构 |
