# 🔍 代码审查指南

## 1. 概述

代码审查是保证代码质量的关键环节。本指南涵盖**接收审查**和**请求审查**两个方向，以及审查清单和评分标准。

## 2. 接收审查的工作流 (superpowers:receiving-code-review)

### 2.1 接收审查流程

```
┌─────────────────────────────────────────────────────────────┐
│                  接收审查工作流                               │
│                                                             │
│  收到审查请求 ──▶ 理解上下文 ──▶ 执行审查 ──▶ 提供反馈        │
│        │              │               │             │       │
│        ▼              ▼               ▼             ▼       │
│    确认范围       了解代码背景    逐项检查      具体可操作    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 审查执行步骤

**步骤1 - 确认审查范围**
- 查看关联的PR/Issue
- 确认要审查的代码变更
- 了解业务背景和目标

**步骤2 - 阅读代码上下文**
- 查看相关文件和测试
- 理解现有代码结构
- 确认代码风格约定

**步骤3 - 执行逐项审查**
- 功能正确性
- 代码质量
- 安全性
- 性能考虑
- 测试覆盖

**步骤4 - 提供建设性反馈**
- 优点先肯定
- 问题要具体
- 建议要可操作

### 2.3 反馈模板

```markdown
## 代码审查反馈

### ✅ 做得好的地方
- [具体指出代码中值得肯定的部分]

### 💡 改进建议
- **问题描述**: 具体指出问题位置和现象
- **建议方案**: 提供可操作的解决方案
- **参考示例**: 如有类似的好实践可参考

### ❌ 必须修复
- **阻断问题**: 影响功能/安全的严重问题
- **修复方式**: 明确的修复指导

### 📝 非阻断备注
- 代码风格偏好
- 优化建议（可选实现）
```

## 3. 请求审查的工作流 (superpowers:requesting-code-review)

### 3.1 请求审查流程

```
┌─────────────────────────────────────────────────────────────┐
│                  请求审查工作流                               │
│                                                             │
│  完成开发 ──▶ 自我审查 ──▶ 准备材料 ──▶ 提交审查            │
│      │            │              │             │             │
│      ▼            ▼              ▼             ▼             │
│  功能实现    预检清单      PR描述       通知审查者          │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 提交前自检清单

**功能完整性**
- [ ] 代码变更符合需求
- [ ] 边界条件已处理
- [ ] 错误处理完善

**代码质量**
- [ ] 语法/类型检查通过
- [ ] Lint检查通过
- [ ] 无调试代码残留

**测试覆盖**
- [ ] 单元测试通过
- [ ] 新增代码有测试覆盖
- [ ] 覆盖率达标

**文档更新**
- [ ] 必要的注释已添加
- [ ] README已更新（如需要）
- [ ] API文档同步更新

### 3.3 PR 描述模板

```markdown
## 变更概述
[简要说明这次变更做什么]

## 变更类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 重构
- [ ] 文档更新
- [ ] 性能优化

## 影响范围
[说明哪些模块/功能会受影响]

## 测试验证
- [ ] 本地测试通过
- [ ] 覆盖率检查通过
- [ ] 手动验证步骤（如有）

## 相关链接
- Issue: [链接]
- 设计文档: [链接]

## 审查注意事项
[特别希望审查者关注的地方]
```

## 4. 审查清单

### 4.1 功能正确性

| 检查项 | 说明 | 优先级 |
|-------|------|-------|
| 逻辑正确 | 代码逻辑符合业务需求 | 必须 |
| 边界处理 | 空值、边界值正确处理 | 必须 |
| 错误处理 | 异常情况有适当处理 | 必须 |
| 并发安全 | 多线程场景安全 | 高 |

### 4.2 代码质量

| 检查项 | 说明 | 优先级 |
|-------|------|-------|
| 命名规范 | 变量/函数命名清晰有意义 | 高 |
| 函数单一职责 | 每个函数只做一件事 | 高 |
| 代码复杂度 | 圈复杂度 ≤ 15 | 高 |
| 重复代码 | 无明显重复逻辑 | 中 |

### 4.3 安全性

| 检查项 | 说明 | 优先级 |
|-------|------|-------|
| 输入验证 | 用户输入经过验证 | 必须 |
| 敏感信息 | 无硬编码密码/密钥 | 必须 |
| 权限检查 | 操作有适当权限验证 | 必须 |
| 依赖安全 | 无已知漏洞依赖 | 高 |

### 4.4 性能

| 检查项 | 说明 | 优先级 |
|-------|------|-------|
| 资源释放 | 及时释放资源 | 高 |
| 循环效率 | 避免不必要的循环 | 中 |
| 缓存利用 | 适当使用缓存 | 中 |
| 数据库查询 | N+1问题避免 | 高 |

### 4.5 可维护性

| 检查项 | 说明 | 优先级 |
|-------|------|-------|
| 代码可读 | 逻辑清晰易读 | 高 |
| 注释适当 | 关键逻辑有注释 | 中 |
| 文档同步 | 文档与代码一致 | 中 |
| 向后兼容 | 变更不破坏现有API | 高 |

## 5. 审查评分标准

### 5.1 评分维度

| 维度 | 权重 | 评分范围 |
|-----|------|---------|
| 功能正确性 | 30% | 1-5 |
| 代码质量 | 25% | 1-5 |
| 安全性 | 20% | 1-5 |
| 性能 | 15% | 1-5 |
| 可维护性 | 10% | 1-5 |

### 5.2 评分定义

| 分数 | 等级 | 说明 |
|-----|------|------|
| 5 | 优秀 | 无需修改，可直接合并 |
| 4 | 良好 | 有小建议，可考虑改进 |
| 3 | 及格 | 有改进空间，建议修改 |
| 2 | 较差 | 有较多问题，需要修改 |
| 1 | 不合格 | 严重问题，必须大改 |

### 5.3 合并条件

| 条件 | 说明 |
|-----|------|
| 必须通过 | 功能正确性 ≥ 3，安全性 ≥ 3 |
| 建议通过 | 总体评分 ≥ 3.5 |
| 必须拒绝 | 存在P0阻断问题 |
| 需讨论 | 存在重大设计分歧 |

## 6. 常见问题与最佳实践

### 6.1 常见问题

| 问题类型 | 典型表现 | 处理方式 |
|---------|---------|---------|
| **过度设计** | 为未来需求预留扩展 | 先解决当前需求 |
| **过早优化** | 复杂优化影响可读性 | 先保证正确，再优化 |
| **职责不清** | 函数承担过多功能 | 拆分到单一职责 |
| **沟通不足** | 审查者不理解设计意图 | 增加上下文说明 |

### 6.2 审查者最佳实践

- 及时响应审查请求
- 审查时保持客观中立
- 反馈具体、可操作
- 区分"必须"和"建议"

### 6.3 被审查者最佳实践

- 保持开放心态接收反馈
- 澄清不清晰的问题
- 解释设计决策的原因
- 感谢审查者的时间

### 6.4 沟通技巧

**✅ 好的反馈**
```markdown
这个函数的命名可能让人误解它在验证输入，
建议改为 `validateAndSanitizeInput`，这样更准确
```

**❌ 不好的反馈**
```markdown
这个命名不好
```
