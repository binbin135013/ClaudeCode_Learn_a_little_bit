# 🗑️ 记忆清理与GC

> ⏱ 难度: ★☆☆ | 重要性: ★★★ | 推荐学习时间: 30分钟
> 💡 **适用场景**: 记忆膨胀、项目交接、跨项目隔离

---

## 概述

Claude Code 的记忆系统采用**文件化存储**，没有传统数据库的自动GC机制。本文档介绍如何主动管理记忆生命周期，避免记忆膨胀和跨项目污染。

### 核心问题

```
为什么需要手动清理记忆？
├── 记忆文件无限增长 → 加载变慢
├── 跨项目记忆混淆 → 上下文污染
└── 过时记忆干扰 → 决策失误
```

---

## 一、记忆文件生命周期

### 1.1 记忆文件类型

| 类型 | 位置 | 清理风险 |
|------|------|---------|
| **Auto Memory** | `~/.claude/projects/<project>/memory/` | 低，可安全清理 |
| **Agent Memory** | `~/.claude/agent-memory/<type>/` | 中，注意scope |
| **Session Memory** | Session 目录 | 高，丢失当前进度 |
| **Team Memory** | 团队共享目录 | 高，涉及多人 |

### 1.2 记忆文件命名约定

```
# 标准格式
MEMORY.md                           # 主索引文件
<category>-<date>.md               # 分类-日期格式
<topic>-<hash>.md                 # 主题-哈希格式

# 实际示例
user-preferences-2024-01-15.md    # 用户偏好
project-context-a3f8b2c.md         # 项目上下文
decision-20240115-143022.md       # 决策记录（带时间戳）
```

### 1.3 文件大小限制

| 文件 | 硬限制 | 软限制（推荐） |
|------|--------|----------------|
| MEMORY.md | 200行 / 25KB | 100行 / 15KB |
| 单个记忆文件 | 无硬限制 | 50KB |
| 全部记忆 | 无限制 | 2MB（项目级） |

---

## 二、自动清理机制

### 2.1 MEMORY.md 自动摘要

当 MEMORY.md 超过限制时，系统自动触发摘要：

```
触发条件
├── 文件超过 200 行
└── 文件超过 25KB

执行流程
1. 创建 .memory/archive/ 目录（如不存在）
2. 移动原文件到 archive/
3. 生成新的 MEMORY.md 摘要
4. 保留关键引用和摘要
```

### 2.2 懒清理策略

| 策略 | 触发条件 | 行为 |
|------|---------|------|
| **即时清理** | 单文件 > 50KB | 警告，建议拆分 |
| **定期清理** | 30天未访问 | 归档到 archive |
| **主动清理** | 人工触发 | 完全清理 |

### 2.3 清理命令

```bash
# 查看记忆统计
/memory stats

# 查看记忆大小
/memory list --size

# 清理过期记忆（30天前）
/memory clean --days 30

# 清理全部记忆（危险！）
/memory clean --all
```

## 三、手动清理指南

### 3.1 项目记忆清理

当项目结束或交接时，执行以下步骤：

```
步骤 1: 备份重要记忆
├── 复制 .claude/projects/<project>/memory/ 到安全位置
└── 保留决策记录和上下文

步骤 2: 清理 MEMORY.md
├── 保留项目名称和目标
├── 删除具体实现细节（已过时）
└── 保留可复用的经验教训

步骤 3: 清理独立记忆文件
├── 删除具体代码路径引用
├── 删除已废弃的配置
└── 保留通用的项目模式
```

### 3.2 用户偏好清理

```bash
# 用户级偏好位置
~/.claude/agent-memory/user/

# 清理策略
├── 保留：长期有效的偏好（代码风格、工作习惯）
├── 删除：项目特定偏好（已结束项目）
└── 归档：不确定的偏好到 archive/
```

### 3.3 跨项目隔离

```
为什么要隔离？
├── 项目A的记忆不应影响项目B
├── 避免上下文污染
└── 保持决策独立性

隔离方法
├── local scope: 仅本机器，不跨设备
├── project scope: 仅本项目，不跨团队
└── user scope: 需要定期审查，删除项目特定内容
```

## 四、TTL管理策略

### 4.1 TTL设计原则

> **"记忆有保质期，过期需复习"**

| 记忆类型 | 推荐TTL | 原因 |
|---------|--------|------|
| **用户偏好** | 永久 | 长期有效 |
| **项目上下文** | 项目周期 | 随项目结束 |
| **决策记录** | 6个月 | 决策可能过时 |
| **技术笔记** | 12个月 | 技术更新快 |
| **团队约定** | 3个月 | 团队可能变化 |

### 4.2 TTL实现方法

在记忆文件 frontmatter 中添加过期日期：

```markdown
---
name: project-context
description: 项目关键上下文
type: project
created: 2024-01-15
expires: 2024-07-15          # TTL: 6个月
tags: [context, project]
---

# 项目上下文

## 项目目标
...
```

### 4.3 TTL检查清单

```markdown
## 季度记忆审查清单

- [ ] 检查所有记忆文件的 expires 字段
- [ ] 更新过期记忆或删除
- [ ] 验证 MEMORY.md 索引是否同步
- [ ] 清理 archive/ 中的归档
- [ ] 记录本次审查日期
```

## 五、跨项目记忆共享

### 5.1 何时共享

| 共享场景 | 推荐程度 | 说明 |
|---------|---------|------|
| **团队代码规范** | ✓✓✓ | 强烈推荐 |
| **通用工具配置** | ✓✓ | 谨慎共享 |
| **项目特定记忆** | ✗ | 不应共享 |
| **个人偏好** | ✗ | 不应共享 |

### 5.2 共享机制

```
共享位置: Team Memory
├── 团队成员可见
├── 版本控制友好
└── 需要明确命名空间

命名规范: team/<team-name>/<category>/
示例:
├── team/backend/standards.md    # 团队规范
├── team/backend/tooling.md       # 团队工具
└── team/backend/decisions.md     # 团队决策
```

### 5.3 共享冲突解决

```
冲突场景
├── 修改同一文件 → 使用 Git 风格冲突解决
├── 删除他人共享记忆 → 需确认 + 通知
└── 过时共享记忆 → 标记 expires + 通知作者

解决流程
1. 发现冲突 → 暂停操作
2. 联系作者 → 确认保留内容
3. 合并修改 → 保留双方有价值内容
4. 记录解决 → 更新冲突解决日志
```

## 六、冲突解决流程

### 6.1 冲突类型

| 冲突类型 | 描述 | 解决方法 |
|---------|------|---------|
| **内容冲突** | 同一记忆有不同版本 | 以更新时间为准 |
| **语义冲突** | 两记忆相互矛盾 | 需人工判断 |
| **重复冲突** | 同一信息多处记录 | 合并 + 删除重复 |
| **归属冲突** | 记忆作者不明确 | 追溯 git blame |

### 6.2 解决流程图

```
发现冲突
    │
    ▼
1. 确定冲突类型
    │
    ├── 内容冲突 → 保留最新 / 合并版本
    ├── 语义冲突 → 咨询相关人
    ├── 重复冲突 → 合并并标记废弃
    └── 归属冲突 → 查看 git 历史
    │
    ▼
2. 记录解决
    │
    ▼
3. 更新 MEMORY.md 索引
    │
    ▼
4. 通知相关方（如涉及团队）
```

### 6.3 冲突解决记录模板

```markdown
## 冲突解决记录

**冲突文件**: user-preferences.md
**发现日期**: 2024-01-20
**冲突类型**: 内容冲突

**解决方式**: 合并版本
**保留内容**: 
- [x] 缩进风格（保留新版本）
- [x] 命名规范（合并两版本）

**废弃内容**:
- [ ] 注释风格（旧版本，已过时）

**解决人**: [name]
**确认人**: [name]
```

## 七、最佳实践清单

### 7.1 日常维护

```
每日
├── 不产生新的临时记忆文件
└── 有价值信息及时固化

每周
├── /memory list 检查记忆状态
└── 清理明显过时的临时记忆

每月
├── 审查 MEMORY.md 大小
├── 更新 expires 字段
└── 归档长期未访问的记忆

每季度
├── 完整记忆审查
├── 清理 archive/
└── 更新团队共享记忆
```

### 7.2 项目交接

```
交接前
├── 备份全部记忆
├── 区分：带走 / 留下 / 删除
└── 编写交接文档

交接时
├── 导出记忆为 Markdown
├── 标注关键记忆位置
└── 说明清理计划

交接后
├── 本地清理项目记忆
├── 确认团队记忆归属
└── 更新 MEMORY.md 索引
```
