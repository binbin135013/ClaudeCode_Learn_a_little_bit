# 🔄 OMC 工作流

## 1. 工作流概览

OMC 提供四大核心工作流，覆盖从简单任务到复杂协作的所有场景：

```
┌─────────────────────────────────────────────────────────┐
│                    OMC 工作流体系                         │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │  autopilot  │  │    ralph    │  │  ultrawork  │     │
│  │  全自主执行  │  │  自我修正   │  │   并行执行  │     │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘     │
│         └────────────────┼────────────────┘            │
│                          ▼                              │
│                   ┌─────────────┐                       │
│                   │    team     │                       │
│                   │   团队协作   │                       │
│                   └─────────────┘                       │
└─────────────────────────────────────────────────────────┘
```

---

## 2. Autopilot - 全自主执行

### 2.1 核心定位

Autopilot 是 OMC 的**端到端全自动执行**工作流，从想法到可运行代码一气呵成。

### 2.2 执行流程

```
用户输入
    │
    ▼
┌─────────────────────────────────────┐
│  1. 理解需求                         │
│     - 解析用户意图                    │
│     - 识别关键约束                    │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│  2. 自动规划                         │
│     - 分解任务步骤                    │
│     - 确定执行顺序                    │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│  3. 代码生成                          │
│     - 调用必要工具                    │
│     - 生成完整实现                    │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│  4. 验证结果                          │
│     - 运行测试                        │
│     - 检查质量                        │
└────────────────┬────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────┐
│  5. 迭代优化 (可选)                   │
│     - 修复问题                        │
│     - 优化代码                        │
└────────────────┬────────────────────┘
                 │
                 ▼
              完成
```

### 2.3 使用示例

```
# 基本用法
autopilot 创建一个用户认证系统

# 指定技术栈
autopilot 使用 Express + JWT 构建 REST API

# 完整功能
autopilot 创建一个博客系统，包含文章发布、评论、用户管理功能
```

### 2.4 适用场景

| 场景 | 示例 | 优势 |
|-----|------|------|
| 快速原型 | "autopilot 做一个天气应用" | 从零到可用快速 |
| 完整功能 | "autopilot 用户管理系统" | 端到端完整实现 |
| 技术验证 | "autopilot 用 Rust 重写这个模块" | 完整技术栈切换 |

### 2.5 注意事项

- 适用于**需求相对明确**的任务
- 复杂需求建议先用 `plan` 或 `deep-interview` 澄清
- 关键业务逻辑建议配合 `ultraqa` 做质量验证

## 3. Ralph - 自我修正循环

### 3.1 核心定位

Ralph 是**持久化循环工作流**，持续执行-验证-修正，直到任务完成或达到目标条件。

### 3.2 执行流程

```
┌──────────────────────────────────────────────────────┐
│                    Ralph 循环                         │
│                                                      │
│    ┌─────────┐    ┌─────────┐    ┌─────────┐        │
│    │  执行   │───▶│  验证   │───▶│  修正   │        │
│    │ Execute │    │ Verify  │    │  Fix   │        │
│    └─────────┘    └─────────┘    └─────────┘        │
│         ▲              │              │              │
│         │              ▼              │              │
│         │         ┌─────────┐         │              │
│         │         │ 通过?   │         │              │
│         │         └───┬─────┘         │              │
│         │             │是            │否            │
│         │             ▼              └──────────────┘
│         │          完成
│         └────────────────────────────────────────────┘
│                          循环
└──────────────────────────────────────────────────────┘
```

### 3.3 使用示例

```
# 持续修复直到通过
ralph 直到所有单元测试通过

# 持续优化直到达标
ralph 持续优化性能直到 API 响应时间低于 100ms

# 持续改进直到满足条件
ralph 直到代码覆盖率超过 80%

# 具体问题修复
ralph 修复登录页面在 Safari 上的兼容性问题
```

### 3.4 退出条件

| 条件类型 | 说明 |
|---------|------|
| **成功退出** | 所有验证通过，达到目标条件 |
| **手动取消** | 用户输入 `/cancel` |
| **超时退出** | 达到最大循环次数 |
| **死循环检测** | 无法取得进展时自动退出 |

### 3.5 适用场景

| 场景 | 示例 | 原因 |
|-----|------|------|
| Bug 修复 | "ralph 修复这个空指针异常" | 需要多轮调试-验证 |
| 性能优化 | "ralph 优化到 100ms 以内" | 迭代调优直到达标 |
| 测试达标 | "ralph 覆盖率到 80%" | 持续补充直到合格 |
| 兼容性问题 | "ralph 修复浏览器兼容" | 多轮调试-验证-修复 |

### 3.6 与 UltraQA 的区别

```
Ralph    → 目标驱动，循环直到满足条件
UltraQA   → 流程驱动，测试-验证-修复-重复
```

## 4. Ultrawork - 并行执行

### 4.1 核心定位

Ultrawork 是**高吞吐量并行执行**工作流，将多个独立任务同时分发执行。

### 4.2 执行流程

```
                    用户输入
                        │
                        ▼
            ┌───────────────────────┐
            │    任务分析           │
            │  - 识别可并行任务      │
            │  - 分析依赖关系        │
            └───────────┬───────────┘
                        │
                        ▼
    ┌─────────────────────────────────────────┐
    │           并行分发                        │
    │                                         │
    │  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
    │  │ Worker1 │  │ Worker2 │  │ Worker3 │ │
    │  │ 重构A   │  │ 重构B   │  │ 重构C   │ │
    │  └────┬────┘  └────┬────┘  └────┬────┘ │
    │       │            │            │       │
    └───────┼────────────┼────────────┼───────┘
            │            │            │
            ▼            ▼            ▼
    ┌─────────────────────────────────────────┐
    │           结果聚合                        │
    │  - 收集所有 Worker 结果                  │
    │  - 合并冲突 (如有)                        │
    │  - 生成汇总报告                          │
    └─────────────────────────────────────────┘
                        │
                        ▼
                     完成
```

### 4.3 使用示例

```
# 并行重构多个文件
ulw 重构 src/ 下的所有 util 模块

# 并行处理多个任务
ultrawork 同时运行构建、测试、部署

# 指定并行度
ulw 并行处理这 10 个模块的重构，使用 5 个 worker

# 并行研究与实现
ultrawork 研究三种缓存方案并实现最优选择
```

### 4.4 并行度配置

| 配置方式 | 说明 |
|---------|------|
| 自动 | OMC 根据任务数自动决定 |
| 手动指定 | `ulw 任务列表 / max 5` |
| 系统限制 | 最大并发数由配置决定 |

### 4.5 适用场景

| 场景 | 示例 | 加速比 |
|-----|------|--------|
| 模块重构 | 10 个独立模块同时重构 | ~5x |
| 批量处理 | 批量转换 100 个文件 | ~10x |
| 构建测试 | 并行构建和测试 | ~2x |
| 研究分析 | 并行研究多个方案 | ~3x |

### 4.6 依赖管理

```
Ultrawork 任务分类:
├── 独立任务    → 直接并行执行
├── 依赖任务    → 按依赖顺序执行
└── 混合任务    → 先独立后依赖
```

## 5. Team - 团队协作

### 5.1 核心定位

Team 是**多 Agent 协调编排**工作流，多个专业化 Agent 共享任务列表、协作完成复杂任务。

### 5.2 团队架构

```
┌─────────────────────────────────────────────────────┐
│                    Team 团队                        │
│                                                     │
│              ┌─────────────────┐                    │
│              │   Coordinator   │                    │
│              │   (协调者)      │                    │
│              └────────┬────────┘                    │
│                       │                             │
│         ┌─────────────┼─────────────┐              │
│         ▼             ▼             ▼              │
│   ┌─────────┐   ┌─────────┐   ┌─────────┐        │
│   │ Agent 1 │   │ Agent 2 │   │ Agent 3 │        │
│   │ Planner │   │Executor │   │ Verifier│        │
│   └─────────┘   └─────────┘   └─────────┘        │
│                                                     │
│   共享: 任务列表、上下文、状态                        │
└─────────────────────────────────────────────────────┘
```

### 5.3 Team Pipeline

```
team-plan → team-prd → team-exec → team-verify → team-fix
```

| 阶段 | 说明 | 产出 |
|-----|------|------|
| `team-plan` | 任务分解和计划 | 任务列表 |
| `team-prd` | 需求评审和确认 | PRD 文档 |
| `team-exec` | 分配并执行任务 | 代码/实现 |
| `team-verify` | 验证结果和质量 | 验证报告 |
| `team-fix` | 修复发现的问题 | 修复代码 |

### 5.4 使用示例

```
# 基本团队协作
team 3 完成这个电商模块的重构

# 指定角色
team 3:planner,executor,verifier 实现支付功能

# 带条件
team 3 直到所有测试通过

# 多模型协作
omc team 3:codex,gemini "研究微服务架构方案"
```

### 5.5 团队角色

| 角色 | Agent | 职责 |
|-----|-------|------|
| 规划者 | planner | 任务分解、优先级排序 |
| 执行者 | executor | 代码实现、功能开发 |
| 审查者 | verifier | 质量验证、测试验证 |
| 架构师 | architect | 系统设计、技术决策 |
| 调试者 | debugger | 问题定位、故障排除 |

### 5.6 通信机制

```javascript
// Team 内部通信
TeamCreate({ name: "重构团队", agents: 3 });

// 发送消息
SendMessage({
  to: "executor-2",
  content: "请接手任务 #5",
  type: "task-assign"
});

// 创建任务
TaskCreate({
  title: "实现用户认证",
  assignee: "executor-1",
  priority: "high"
});

// 更新任务
TaskUpdate({
  id: "task-5",
  status: "completed"
});
```

### 5.7 适用场景

| 场景 | 说明 | 团队规模 |
|-----|------|---------|
| 大型重构 | 多模块协同重构 | 3-5 Agent |
| 新功能开发 | 前后端分离开发 | 2-3 Agent |
| 系统设计 | 复杂系统架构设计 | 3 Agent |
| 代码审查 | 多维度审查 | 2 Agent |

## 6. 工作流对比与选择

### 6.1 四象限选择图

```
        复杂度
          ↑
          │
    架构/ │   architect    team + ralph
   深度   │
   分析   │   planner       team
          │
          ├───────────────────────────→
          │   autopilot      ultrawork
   简单/  │    单步执行       并行执行
   快速   │
          └───────────────────────────>
              单Agent          多Agent
                          协作度
```

### 6.2 选择矩阵

| 任务特征 | 推荐工作流 | 原因 |
|---------|-----------|------|
| 需求明确，从零实现 | autopilot | 端到端自动化 |
| 需要多轮迭代优化 | ralph | 持续修正 |
| 多个独立任务 | ultrawork | 高吞吐并行 |
| 复杂任务分工 | team | 多 Agent 协作 |
| 测试驱动开发 | ultraqa | QA 循环 |
| 多模型综合分析 | ccg | 三模型优势 |

### 6.3 组合使用

```
大型复杂任务的工作流组合:

1. plan / deep-interview
   └── 澄清需求，制定计划

2. team N
   ├── Agent 1: autopilot 实现模块 A
   ├── Agent 2: autopilot 实现模块 B
   └── Agent 3: 持续集成配置

3. ralph
   └── 持续优化直到所有指标达标

4. ultraqa
   └── 最终质量验证

5. verifier
   └── 确认完成，准备交付
```
