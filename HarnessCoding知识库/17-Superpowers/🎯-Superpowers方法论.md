# 🎯 Superpowers 方法论

## 概述

Superpowers 方法论是 7 个核心 Skill 背后的指导思想，包括执行机制、质量原则和设计规范。

---

## 1. HARD-GATE 机制

### 1.1 什么是 HARD-GATE

HARD-GATE 是 Superpowers 的**核心执行机制**，确保每个开发阶段都被正确执行后才进入下一阶段。

```
┌─────────────────────────────────────────────────┐
│                  HARD-GATE 流程                  │
├─────────────────────────────────────────────────┤
│                                                 │
│  ┌───┐                                          │
│  │ H │ Halt - 在每个阶段门口停止                  │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ A │ Assert - 断言前置条件满足                  │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ R │ Review - 审查本阶段成果                    │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ D │ Decide - 决定是否可以进入下一阶段          │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ G │ Guarantee - 保证流程完整性                 │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ A │ Automate - 自动化执行检查                  │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ T │ Trace - 追踪每个决策                      │
│  └───┘                                          │
│    │                                            │
│    ▼                                            │
│  ┌───┐                                          │
│  │ E │ Ensure - 确保结果质量                     │
│  └───┘                                          │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 1.2 各阶段详解

#### H - Halt (停止)

**含义**: 在进入下一阶段之前，当前阶段必须完全结束。

**实践**:
- 不要跳过任何阶段
- 不要并行处理应该串行的阶段
- 在阶段门口停下来检查

#### A - Assert (断言)

**含义**: 明确并验证前置条件是否满足。

**实践**:
- 进入下一阶段前，验证本阶段的产出
- 使用检查清单
- 记录验证结果

#### R - Review (审查)

**含义**: 审查本阶段的成果，确保质量。

**实践**:
- 代码审查
- 设计审查
- 方案审查

#### D - Decide (决定)

**含义**: 基于审查结果，决定是否可以继续。

**实践**:
- 通过 → 进入下一阶段
- 不通过 → 修复后重新审查

#### G - Guarantee (保证)

**含义**: 保证整个流程的完整性和可追溯性。

**实践**:
- 记录所有决策
- 保留审计日志
- 确保无遗漏步骤

#### A - Automate (自动化)

**含义**: 将检查自动化，减少人工失误。

**实践**:
- CI/CD 自动化测试
- Lint 自动化
- 代码质量检查自动化

#### T - Trace (追踪)

**含义**: 追踪每个决策的原因和结果。

**实践**:
- 决策日志
- 变更追踪
- 问题溯源

#### E - Ensure (确保)

**含义**: 确保最终结果满足需求。

**实践**:
- 最终验证
- 质量门禁
- 交付确认

### 1.3 HARD-GATE 在各 Skill 中的应用

#### TDD 中的 HARD-GATE

```
红(Green) ──▶ Assert: 测试失败
     │              │
     ▼              ▼
  Assert: 代码有效  │
     │              │
     ▼              ▼
  Review: 代码审查  │
     │              │
     ▼              ▼
  Decide: 合并? ────┘
```

#### Debugging 中的 HARD-GATE

```
复现 ──▶ Assert: 问题已复现
   │           │
   ▼           ▼
定位 ──▶ Assert: 根因已找到
   │           │
   ▼           ▼
修复 ──▶ Review: 代码审查
   │           │
   ▼           ▼
验证 ──▶ Decide: 问题已解决?
```

## 2. "Even 1% chance means invoke" 原则

### 2.1 原则定义

> 只要有任何可能性会出现问题，就应该调用对应的 Skill 来预防或处理。

这不是过度工程，而是一种**风险管理策略**。

### 2.2 决策矩阵

| 风险场景 | 可能性 | 是否调用 | 调用的 Skill |
|---------|-------|---------|-------------|
| 新功能开发 | 任何 | 是 | `tdd` |
| 复杂算法 | >1% | 是 | `brainstorming` |
| Bug 修复 | 任何 | 是 | `systematic-debugging` |
| 大型任务 | >1% | 是 | `subagent-driven-development` |
| 代码变更 | 任何 | 是 | `receiving-code-review` |

### 2.3 触发决策树

```
是否存在风险?
│
├── Bug 风险 ──────────────────▶ systematic-debugging
│
├── 设计不清晰 ────────────────▶ brainstorming
│
├── 代码质量问题 ──────────────▶ receiving-code-review
│
├── 大型/复杂任务 ─────────────▶ subagent-driven-development
│
├── 新功能开发 ───────────────▶ tdd
│
└── 无明显风险 ────────────────▶ 直接开发
```

### 2.4 实践指导

**不要犹豫调用 Skill**:
- 调用 Skill 的成本远低于修复问题
- Skill 是轻量级的，不会显著增加工作量
- 预防胜于治疗

**常见误区**:
1. "这个问题太简单，不需要 Skill" → 可能浪费时间在简单问题上
2. "调用 Skill 太麻烦了" → 实际上 Skill 是为了提高效率
3. "我知道怎么做，不需要规划" → 可能遗漏重要细节

## 3. Skill 元数据规范

### 3.1 必需字段

```yaml
---
# 必需字段
name: skill-name              # Skill 唯一标识符
type: skill                   # 类型固定为 skill
description: 技能描述         # 一行描述技能功能
---
```

### 3.2 可选字段

```yaml
---
# 触发配置
trigger:
  - "trigger-word-1"
  - "trigger-word-2"

# 版本和作者
version: "1.0.0"
author: "Author Name"

# 标签分类
tags:
  - tag1
  - tag2

# Agent 配置
agent:
  model: sonnet
  temperature: 0.7
  max_tokens: 4096

# 工作流配置
workflow:
  parallel: false
  retry: 3
  timeout: 300000

# 依赖配置
requires:
  tools: [Read, Edit, Bash]
  skills: [other-skill]
---
```

### 3.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|-----|-----|-----|-----|
| `name` | string | 是 | Skill 唯一标识 |
| `type` | string | 是 | 固定为 "skill" |
| `description` | string | 是 | 功能描述 |
| `trigger` | array | 否 | 触发词列表 |
| `version` | string | 否 | 版本号 |
| `author` | string | 否 | 作者 |
| `tags` | array | 否 | 标签 |
| `agent` | object | 否 | Agent 配置 |
| `workflow` | object | 否 | 工作流配置 |
| `requires` | object | 否 | 依赖配置 |

## 4. TDD 铁律

### 4.1 三条铁律

```
TDD 铁律

1. 除非是为了使一个失败的测试通过，否则不允许编写任何产品代码

2. 不允许编写任何多余的产品代码，恰好让测试通过即可

3. 不允许编写任何多余的测试，只有让当前失败测试通过后才能停止
```

### 4.2 铁律解读

#### 铁律 1：先写测试

**含义**: 在编写任何产品代码之前，必须先编写一个失败的单元测试。

**违反示例**:
```javascript
// 错误：先写了产品代码
function add(a, b) {
  return a + b;
}
test('加法', () => {
  expect(add(1, 2)).toBe(3);
});

// 正确：先写测试
test('加法', () => {
  expect(add(1, 2)).toBe(3);
});
function add(a, b) {
  return a + b;
}
```

#### 铁律 2：最少代码

**含义**: 只需编写让测试通过的代码，不要过度设计。

**违反示例**:
```javascript
// 错误：编写了超出需求的代码
test('加法', () => {
  expect(add(1, 2)).toBe(3);
});
function add(a, b) {
  // 错误：实现了减法，但测试不需要
  return a + b - b;
}

// 正确：只写需要的代码
test('加法', () => {
  expect(add(1, 2)).toBe(3);
});
function add(a, b) {
  return a + b;
}
```

#### 铁律 3：不多余测试

**含义**: 只编写让当前失败测试通过所需的测试，不要预先编写未来的测试。

**违反示例**:
```javascript
// 错误：编写了不需要的测试
test('加法', () => {
  expect(add(1, 2)).toBe(3);
});
test('减法', () => {  // 这个测试当前不需要
  expect(subtract(3, 1)).toBe(2);
});

// 正确：只写当前需要的测试
test('加法', () => {
  expect(add(1, 2)).toBe(3);
});
```

### 4.3 TDD 循环

```
         ┌────────────────────────────────────┐
         │                                    │
         ▼                                    │
    ┌─────────┐      ┌─────────┐              │
    │   红    │ ───▶ │   绿    │              │
    │ (失败)  │      │ (通过)  │              │
    └─────────┘      └─────────┘              │
         ▲                                    │
         │                                    │
         │          ┌─────────┐               │
         └───────── │  重构   │               │
                    │(改善)   │               │
                    └─────────┘               │
                              │                │
                              └────────────────┘
                                    重复
```

### 4.4 TDD 优势

| 优势 | 说明 |
|-----|------|
| **设计驱动** | 测试先写促进好设计 |
| **文档化** | 测试即文档 |
| **安全感** | 重构时不怕破坏功能 |
| **快速反馈** | 立即知道代码是否正确 |
| **减少调试** | 减少 debug 时间 |

### 4.5 TDD 常见误区

| 误区 | 正确做法 |
|-----|---------|
| 为了覆盖率写测试 | 为了验证行为写测试 |
| 测试写得过于详细 | 测试保持简洁 |
| 不写集成测试 | 平衡单元测试和集成测试 |
| 跳过重构阶段 | 每次循环都重构 |
| 测试实现而非行为 | 测试输入输出，不是实现 |
