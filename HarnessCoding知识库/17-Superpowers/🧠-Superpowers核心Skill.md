# 🧠 Superpowers 核心 Skill

## 概述

Superpowers 提供 7 个核心 Skill，涵盖开发全流程：

| Skill | 方法论 | 核心价值 |
|-------|--------|---------|
| brainstorming | 头脑风暴与设计 | 明确需求，避免返工 |
| systematic-debugging | 四阶段调试法 | 高效定位问题 |
| test-driven-development | TDD 红绿重构 | 保证代码质量 |
| subagent-driven-development | 子代理驱动开发 | 并行加速任务 |
| receiving-code-review | 接收代码审查 | 持续改进代码 |
| requesting-code-review | 请求代码审查 | 规范化协作 |
| writing-skills | Skill 创作方法论 | 扩展能力边界 |

---

## 1. brainstorming - 头脑风暴与设计

### 1.1 核心理念

> 在编码之前，先想清楚。

大多数开发返工源于设计阶段的不完善。brainstorming Skill 帮助你在动手前系统性地思考问题。

### 1.2 触发方式

```bash
brainstorm
/skill brainstorming
```

### 1.3 工作流程

```
┌─────────────────────────────────────────┐
│            brainstorming 工作流          │
├─────────────────────────────────────────┤
│  1. 问题定义                             │
│     - 明确要解决的核心问题                 │
│     - 定义成功标准                        │
│                                         │
│  2. 需求分析                             │
│     - 功能性需求                          │
│     - 非功能性需求                         │
│     - 约束条件                            │
│                                         │
│  3. 方案探索                             │
│     - 列出所有可能的方案                   │
│     - 每个方案列出优点和缺点               │
│                                         │
│  4. 方案评估                             │
│     - 使用决策矩阵                        │
│     - 权衡利弊                            │
│                                         │
│  5. 决策总结                             │
│     - 确定最终方案                        │
│     - 明确实施计划                        │
└─────────────────────────────────────────┘
```

### 1.4 决策矩阵示例

| 方案 | 开发时间 | 维护成本 | 扩展性 | 安全性 | 总分 |
|-----|---------|---------|-------|-------|------|
| 方案 A | 3 | 2 | 4 | 5 | 14 |
| 方案 B | 5 | 4 | 3 | 4 | 16 |
| 方案 C | 2 | 5 | 2 | 3 | 12 |

### 1.5 使用示例

```
用户: brainstorm 设计一个实时聊天系统

输出:
## 头脑风暴：实时聊天系统设计

### 1. 问题定义
- 核心：实现低延迟的实时消息传递
- 成功标准：消息延迟 < 200ms

### 2. 需求分析
- 功能：用户认证、房间管理、消息发送、在线状态
- 非功能：实时性、可扩展、可靠性
- 约束：需要支持移动端

### 3. 方案探索
方案 A: WebSocket + Redis Pub/Sub
方案 B: Server-Sent Events + Kafka
方案 C: 第三方服务 (Pusher/Ably)

### 4. 方案评估
...

### 5. 决策
选择方案 A，原因：...
```

### 1.6 适用场景

- 新项目启动
- 大型功能设计
- 技术选型
- 架构决策

## 2. systematic-debugging - 四阶段调试法

### 2.1 核心理念

> 结构化调试，拒绝盲目试错。

调试是开发过程中最耗时的任务之一。systematic-debugging 提供系统化的调试方法。

### 2.2 触发方式

```bash
systematic-debugging
/skill systematic-debugging
```

### 2.3 四阶段工作流

```
┌─────────────────────────────────────────┐
│       systematic-debugging 四阶段        │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────┐    ┌─────────┐            │
│  │ Stage 1 │    │ Stage 2 │            │
│  │  复现   │───▶│  定位   │            │
│  └─────────┘    └─────────┘            │
│       │              │                 │
│       │              ▼                 │
│       │         ┌─────────┐            │
│       │         │ Stage 3 │            │
│       │         │  修复   │            │
│       │         └─────────┘            │
│       │              │                 │
│       ▼              ▼                 │
│  ┌─────────┐    ┌─────────┐            │
│  │ Stage 4 │    │  验证   │            │
│  │  记录   │◀───│         │            │
│  └─────────┘    └─────────┘            │
│                                         │
└─────────────────────────────────────────┘
```

### 2.4 各阶段详解

#### Stage 1: 复现 (Reproduce)

**目标**: 稳定复现问题

**检查清单**:
- [ ] 确定触发条件
- [ ] 收集错误信息
- [ ] 记录复现步骤
- [ ] 验证问题可重复

**关键问题**:
- 这个问题每次都会出现吗？
- 什么情况下会出现？
- 是否与特定数据相关？

#### Stage 2: 定位 (Localize)

**目标**: 缩小范围，找到根因

**检查清单**:
- [ ] 收集相关日志
- [ ] 分析错误堆栈
- [ ] 隔离问题代码
- [ ] 识别根本原因

**定位技巧**:
- 二分查找法
- 添加日志/断点
- 排除法
- 对比分析法

#### Stage 3: 修复 (Fix)

**目标**: 实施解决方案

**检查清单**:
- [ ] 制定修复方案
- [ ] 评估影响范围
- [ ] 实施修复
- [ ] 更新相关测试

**修复原则**:
- 最小改动
- 不引入新 Bug
- 保持代码风格一致

#### Stage 4: 验证 (Verify)

**目标**: 确认问题已解决

**检查清单**:
- [ ] 原问题不再出现
- [ ] 相关功能正常
- [ ] 无回归问题
- [ ] 更新文档（如需要）

### 2.5 使用示例

```
用户: systematic-debugging 用户无法登录

输出:
## 调试：用户无法登录

### Stage 1: 复现
- 收集错误：Invalid credentials
- 复现步骤：使用正确密码尝试登录
- 结果：失败，提示凭据无效

### Stage 2: 定位
- 检查用户表：用户存在
- 检查密码哈希：数据库中的哈希值与预期不符
- 根因：密码更新时哈希算法未正确应用

### Stage 3: 修复
- 重新哈希正确密码
- 更新数据库中的密码字段
- 验证：用户可正常登录

### Stage 4: 验证
- 原问题已解决
- 其他用户登录正常
- 无回归问题
```

### 2.6 适用场景

- Bug 修复
- 错误分析
- 问题溯源
- 异常处理

## 3. test-driven-development - TDD 红绿重构

### 3.1 核心理念

> 红、绿、重构，循环不止。

TDD 是一种开发方法论，通过先写测试再写代码的方式，确保代码质量。

### 3.2 触发方式

```bash
tdd
/skill test-driven-development
```

### 3.3 红绿重构工作流

```
┌─────────────────────────────────────────┐
│           TDD 红绿重构循环               │
├─────────────────────────────────────────┤
│                                         │
│    ┌─────────┐                         │
│    │   红    │  编写一个会失败的测试     │
│    │ (Red)   │                         │
│    └────┬────┘                         │
│         │                               │
│         ▼                               │
│    ┌─────────┐                         │
│    │   绿    │  编写最少的代码使测试通过  │
│    │ (Green) │                         │
│    └────┬────┘                         │
│         │                               │
│         ▼                               │
│    ┌─────────┐                         │
│    │  重构   │  改善代码结构            │
│    │(Refactor)│  保持测试通过            │
│    └────┬────┘                         │
│         │                               │
│         └───────────────────────────────┘
│              继续下一个测试              │
└─────────────────────────────────────────┘
```

### 3.4 TDD 铁律

1. **先写测试，再写代码** - 在编写任何产品代码之前，必须先编写一个失败的单元测试
2. **只写最少的代码** - 只需让测试通过，不要过度设计
3. **立即重构** - 测试通过后，立即改善代码结构

### 3.5 三条规则

```
1. 除非是为了使一个失败的测试通过，否则不允许编写任何产品代码
2. 不允许编写任何多余的产品代码，恰好让测试通过即可
3. 不允许编写任何多余的测试，只有让当前失败测试通过后才能停止
```

### 3.6 使用示例

**目标**: 实现一个计算器

**步骤 1: 红（写测试）**

```javascript
// calculator.test.js
test('加法', () => {
  const calc = new Calculator();
  expect(calc.add(1, 2)).toBe(3);
});
```

**步骤 2: 绿（写代码）**

```javascript
// calculator.js
class Calculator {
  add(a, b) {
    return a + b;
  }
}
```

**步骤 3: 重构**

```javascript
// 重构：添加更多方法
class Calculator {
  add(a, b) {
    return a + b;
  }
  subtract(a, b) {
    return a - b;
  }
}
```

### 3.7 适用场景

- 新功能开发
- 代码重构
- 关键模块实现
- Bug 修复后的测试补充

## 4. subagent-driven-development - 子代理驱动开发

### 4.1 核心理念

> 并行执行，加速任务完成。

当面临大型任务时，将其分解为多个子任务，由子代理并行执行。

### 4.2 触发方式

```bash
subagent
/skill subagent-driven-development
```

### 4.3 工作流程

```
┌─────────────────────────────────────────┐
│      subagent-driven-development        │
├─────────────────────────────────────────┤
│                                         │
│  主代理                                 │
│    │                                   │
│    ├──▶ 子代理 1 (并行) ──┐             │
│    ├──▶ 子代理 2 (并行) ──┼──▶ 结果聚合  │
│    └──▶ 子代理 3 (并行) ──┘             │
│                                         │
└─────────────────────────────────────────┘
```

### 4.4 任务分解原则

| 原则 | 说明 |
|-----|------|
| **独立** | 子任务相互独立，无依赖 |
| **可并行** | 可以同时执行 |
| **单一职责** | 每个子任务只做一件事 |
| **明确边界** | 输入输出清晰 |

### 4.5 使用示例

```
用户: subagent 为博客系统创建 CRUD API

输出:
## 子代理驱动开发：博客 CRUD API

### 任务分解
1. 子代理 1: User 相关 API
2. 子代理 2: Post 相关 API
3. 子代理 3: Comment 相关 API

### 并行执行
[Agent 1] 创建 user create/read/update/delete
[Agent 2] 创建 post create/read/update/delete
[Agent 3] 创建 comment create/read/delete

### 结果聚合
- 整合所有 API
- 统一路由结构
- 验证完整性
```

### 4.6 适用场景

- 大型项目开发
- 多模块同步开发
- 并行任务处理
- 加速开发进度

## 5. receiving-code-review - 接收代码审查

### 5.1 核心理念

> 开放心态，持续改进。

代码审查是提升代码质量的重要手段。receiving-code-review 帮助开发者正确处理审查反馈。

### 5.2 触发方式

```bash
receive-review
/skill receiving-code-review
```

### 5.3 工作流程

```
┌─────────────────────────────────────────┐
│      receiving-code-review 工作流        │
├─────────────────────────────────────────┤
│                                         │
│  1. 接受反馈                             │
│     - 保持开放心态                        │
│     - 不要防御                           │
│     - 记录所有反馈                        │
│                                         │
│  2. 分类处理                             │
│     - 必改项 vs 可选项                    │
│     - 风格 vs 逻辑                       │
│     - 重大 vs 轻微                        │
│                                         │
│  3. 逐项修复                             │
│     - 按优先级处理                        │
│     - 有疑问及时沟通                       │
│                                         │
│  4. 确认完成                             │
│     - 回复审查者                         │
│     - 说明处理结果                        │
│                                         │
└─────────────────────────────────────────┘
```

### 5.4 反馈分类

| 类型 | 说明 | 处理方式 |
|-----|------|---------|
| **Bug** | 功能性问题 | 必须修复 |
| **安全** | 安全漏洞 | 必须修复 |
| **风格** | 代码风格 | 可选择接受 |
| **优化** | 性能改进 | 视情况接受 |
| **疑问** | 需要澄清 | 及时沟通 |

### 5.5 使用示例

```
用户: receive-review 收到以下反馈

反馈列表:
1. [必改] 变量命名不够清晰
2. [必改] 缺少错误处理
3. [可选] 可以使用 const 替代 let
4. [疑问] 这个函数为什么要这样做？

输出:
## 处理代码审查反馈

### 1. 接受反馈
已记录所有 4 条反馈

### 2. 分类
- 必改: 2 项
- 可选: 1 项
- 疑问: 1 项

### 3. 修复
- [x] 重命名变量为更有意义的名称
- [x] 添加 try-catch 错误处理
- [ ] 接受建议，使用 const

### 4. 沟通
针对疑问项：解释设计意图...

### 5. 确认完成
已回复审查者，所有必改项已修复
```

### 5.6 适用场景

- 审查反馈处理
- 代码改进
- 团队协作

```

│     requesting-code-review 工作流         │
├─────────────────────────────────────────┤
│                                         │
│  1. 自我检查                             │
│     - 运行测试                           │
│     - 检查代码风格                        │

## 7. writing-skills - Skill 创作方法论

### 7.1 核心理念

> 封装工作流，扩展能力边界。

将重复的工作流封装为 Skill，实现能力复用。

### 7.2 触发方式

```bash
write-skills
/skill writing-skills
```

### 7.3 Skill 文件结构

```markdown
---
name: skill-name
description: 技能描述
type: skill
trigger:
  - "trigger-word"
version: "1.0.0"
---

# Skill 标题

## 功能描述
详细说明技能的功能

## 触发方式
- 命令: /skill skill-name
- 触发词: trigger-word

## 参数说明
| 参数 | 必填 | 说明 |
|-----|-----|-----|

## 工作流程
1. 步骤一
2. 步骤二
3. 步骤三

## 示例
```bash
trigger-word 参数
```
```

### 7.4 Skill 元数据规范

| 字段 | 必填 | 说明 |
|-----|-----|-----|
| `name` | 是 | Skill 唯一标识 |
| `type` | 是 | 固定为 "skill" |
| `description` | 是 | 功能描述 |
| `trigger` | 否 | 触发词列表 |
| `version` | 否 | 版本号 |

### 7.5 适用场景

- 创建新 Skill
- 封装工作流
- 团队 Skill 共享
- 能力扩展
