# omx (oh-my-codex) 项目解析

## 1. 项目概述

**omx (oh-my-codex)** 是一个专注于 **AI 工作流编排层** 的开源项目，旨在自动化代码重写与迁移的全流程。与单一的代码执行工具不同，omx 将整个重写过程视为一个需要精心编排的复杂工作流，通过 AI 原生的方式协调多个阶段的任务。

## 2. 核心定位

**编排整个重写过程的 AI 工作流层**

omx 的核心价值在于：

- **流程化思维**：将代码迁移拆解为可管理的工作流阶段
- **自动化编排**：减少人工干预，实现端到端的自动化
- **可观测性**：提供工作流执行状态的清晰可见性
- **可扩展性**：支持自定义阶段和插件式组件

## 3. 与 oh-my-openagent 的区别

| 维度 | omx | oh-my-openagent |
|------|-----|-----------------|
| **核心职责** | 工作流编排层 | Agent 执行框架 |
| **专注场景** | 重写/迁移场景 | 通用 Agent 能力 |
| **抽象层次** | 流程编排 | Agent 行为定义 |
| **典型用途** | 代码迁移流水线 | 多 Agent 协作系统 |

简言之：**omx 回答"如何组织"的问题，oh-my-openagent 回答"谁来执行"的问题。**

## 4. 工作流编排设计

omx 采用**阶段门禁模式**（Phase-Gate Pattern）来组织代码迁移：

```
┌─────────────────────────────────────────────────────────┐
│                    迁移工作流 Pipeline                    │
├──────────┬──────────┬──────────┬──────────┬─────────────┤
│  源码分析 │ 依赖映射  │  转换规划  │  分批执行  │   验证通过  │
│  Phase 1 │  Phase 2 │  Phase 3 │  Phase 4 │   Phase 5   │
└──────────┴──────────┴──────────┴──────────┴─────────────┘
```

### 各阶段职责

1. **源码分析** - 解析原始代码结构、识别关键模式
2. **依赖映射** - 建立模块间依赖关系图谱
3. **转换规划** - 生成迁移路径和优先级策略
4. **分批执行** - 按计划逐批执行代码转换
5. **验证通过** - 自动化测试与一致性校验

## 5. 架构特点

### 工作流层 vs 执行层

```
┌────────────────────────────────────┐
│         omx 工作流层                │  ← 编排决策
│   (Orchestration Layer)           │
├────────────────────────────────────┤
│      oh-my-openagent 执行层         │  ← 具体执行
│      (Execution Layer)            │
├────────────────────────────────────┤
│         底层 AI Model              │  ← 智能能力
│      (Claude/Codex/Gemini)         │
└────────────────────────────────────┘
```

**关键设计原则**：

- **工作流层不直接操作代码** - 负责决策和调度
- **执行层专注于任务完成** - 接收指令并产出结果
- **清晰的关注点分离** - 便于独立演进和测试

## 6. 适用场景

### 何时使用 omx

- **大规模代码库迁移** - 需要分批、分阶段进行的重写项目
- **多模块依赖复杂的迁移** - 需要完整的依赖分析和规划
- **需要可观测性的迁移过程** - 团队需要对迁移进度有清晰了解
- **标准化迁移流程** - 希望将迁移经验固化为可复用流程

### 何时考虑替代方案

- **小范围代码修改** - 单文件或少量文件的简单修改
- **即时交互式重写** - 需要快速反馈的人工协作场景
- **通用 Agent 任务** - 非迁移相关的 AI 辅助任务
