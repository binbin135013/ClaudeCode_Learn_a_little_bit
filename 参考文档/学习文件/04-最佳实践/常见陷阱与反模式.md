# 常见陷阱与反模式

> [!summary] 一句话定义
> 收集 Claude Code 和 OpenClaw 使用中**最常踩的坑**，每个都附带错误示范、正确做法和相关链接，帮你少走弯路。

> [!note] 内容性质
> 本文件中的陷阱和建议基于 Claude Code 和 OpenClaw 的使用经验总结。具体的量化数据（Token 消耗、成本估算）为粗略估计，非精确测量。问题描述和解决方案是合理推断，旨在帮助识别常见问题模式。

## Claude Code 陷阱

### 1. CLAUDE.md 过度膨胀

> [!danger] 严重程度：高
> CLAUDE.md 超过 5000 字符，每次请求浪费大量 Token，还会挤占对话上下文空间。

```
# 错误示范：把所有规范塞进一个文件
CLAUDE.md → 12,000 字符（包含完整的代码规范、架构说明、部署流程...）
每次对话额外消耗大量 Token 读取，一个项目累计浪费可观成本。
```

**正确做法**：精简到 2000 字以内，用分层加载。

```markdown
# CLAUDE.md（精简版，< 2000 字符）
## 代码规范
- TypeScript 严格模式，禁止 any
- 详见 docs/ai-context/coding-standards.md

## 架构
- 分层架构，详见 docs/ai-context/architecture.md
```

- 详细方法 → [[CLAUDE-md编写指南]]
- 成本影响 → [[性能与成本优化#精简 CLAUDE.md]]

### 2. --dangerously-skip-permissions 成瘾

> [!danger] 严重程度：高
> 不配白名单就直接跳过权限检查，AI 可能删除文件、修改系统配置。跳过权限是高风险操作。

```bash
# 错误示范
claude --dangerously-skip-permissions
# 完全放弃权限检查，AI 可以执行任何命令

# 正确做法：配合 allowedTools 精确放行
claude  # 默认模式，每步确认
```

**正确做法**：在 `settings.json` 中配置 `allowedTools`，只放行安全的常用操作。

```json
{
  "allowedTools": [
    "Read", "Glob", "Grep",
    "Bash(npm test)",
    "Bash(npm run lint)"
  ]
}
```

- 详细配置 → [[权限与沙箱]]

### 3. Hooks 里写复杂逻辑

> [!warning] 严重程度：中
> Hook 脚本里写 100 行业务逻辑，出问题时难以排查，调试信息淹没在 Claude Code 日志中。

```json
// 错误示范：Hook 内嵌复杂逻辑
{
  "hook": "post-commit",
  "command": "bash -c '...50行内联脚本...'"
}
```

**正确做法**：Hook 只做调度，逻辑放独立脚本。

```json
{
  "hook": "post-commit",
  "command": "bash .claude/hooks/notify-deploy.sh"
}
```

```bash
# .claude/hooks/notify-deploy.sh
#!/bin/bash
# 单一职责：构建成功后发送通知
curl -X POST "$WEBHOOK_URL" -d "构建成功"
```

- 详细说明 → [[Hooks系统]]

### 4. Skills 不用 fork 模式

> [!warning] 严重程度：中
> Skill 的上下文模式选择不当可能导致主对话上下文被撑爆。

```json
// 错误示范：不当使用 shared 模式（上下文膨胀）
{
  "name": "code-review",
  "mode": "shared"
}
```

**正确做法**：推荐使用 `fork` 模式（独立上下文，不影响主会话），只在明确需要访问主会话上下文时才用 `shared`。

```json
{
  "name": "code-review",
  "mode": "fork"  // 隔离上下文，结果摘要后回传
}
```

- 详细说明 → [[Skills开发]]

### 5. MCP 服务器装太多

> [!warning] 严重程度：中
> 配置 10+ 个 MCP Server，每次启动都要初始化所有连接，启动时间从 3 秒变成 30+ 秒，Token 消耗也显著增加。

```
# 错误示范：装了一堆用不上的 MCP Server
.mcp.json → 12 个 server（数据库、GitHub、Slack、Notion、Figma...）
启动耗时大幅增加，每次对话额外消耗可观 Token 用于描述可用工具。
```

**正确做法**：只装当前项目需要的 3-5 个 Server，按项目切换。

```json
{
  "mcpServers": {
    "filesystem": { ... },
    "github": { ... },
    "postgres": { ... }
  }
}
```

- 详细说明 → [[MCP协议]]

### 6. Subagent 递归调用

> [!danger] 严重程度：高
> Agent A 调用 Agent B，Agent B 又调用 Agent A，形成无限循环，迅速耗尽 Token 和 API 配额。

```
# 错误示范：循环依赖
code-reviewer → 调用 executor → 调用 code-reviewer → ...
Token 消耗：每轮大量消耗，几轮后就超限。
```

**正确做法**：设计 DAG（有向无环图）依赖，确保调用链没有环路。

```
# 正确示范：分层 DAG
planner → executor → verifier（单向流动，无环路）
code-reviewer → 只读，不触发其他 agent
```

- 详细说明 → [[Subagent子代理]]

### 7. Git diff 盲目接受

> [!danger] 严重程度：高
> 不审查 AI 生成的 diff 直接 commit，可能引入安全漏洞、破坏性改动或低质量代码。

```bash
# 错误示范
claude "重构认证模块"
git add -A
git commit -m "refactor auth"  # 没有审查任何一行改动
```

**正确做法**：先审查再提交，尤其是涉及权限、认证、数据库的改动。

```bash
# 正确做法
claude "重构认证模块"
git diff          # 人工审查改动
claude /review    # 让 AI 先做代码审查
# 确认无误后再提交
git add src/auth/
git commit
```

- 详细流程 → [[项目工作流]]

### 8. 忽略 .claudeignore

> [!warning] 严重程度：中
> 不配置 `.claudeignore`，AI 会读取 `node_modules`、构建产物、大型日志等无用文件，浪费大量 Token。

```
# 错误：没有 .claudeignore
AI 读取范围 → 整个项目（包括 node_modules/ 等大量无用文件）
Token 浪费：每次对话额外消耗大量 Token 读取无用内容。
```

**正确做法**：配置排除规则，和 `.gitignore` 思路一致。

```
# .claudeignore
node_modules/
dist/
build/
*.log
*.min.js
*.min.css
.env*
```

### 9. 上下文不清就启动任务

> [!tip] 常见新手错误
> 直接让 Claude Code "帮我优化整个项目"，没有给出具体文件、具体问题、具体目标。AI 要么无从下手，要么大范围改动引发问题。

```bash
# 错误示范（太模糊）
claude "优化项目性能"

# 正确示范（具体明确）
claude "src/api/users.ts 的 fetchUsers 函数每次都全量加载，
请改成分页查询，每页 20 条，使用 Prisma 的 skip/take"
```

---

### 10. 滥用 compact 代替 /clear

> [!warning] 严重程度：低
> 上下文快满时频繁使用 `/compact` 压缩，压缩后的信息质量下降，AI 可能遗忘前面的关键决策。

**正确做法**：一个大任务完成后直接 `/clear` 开始新会话，只在需要保留上下文时才用 `/compact`。

## OpenClaw 陷阱

### 1. MEMORY.md 当日志用

> [!danger] 严重程度：高
> 每天往 MEMORY.md 堆内容，膨胀到 10K+ 字符。MEMORY 每次对话都会被加载，导致 Token 消耗剧增。

```
# 错误示范
MEMORY.md:
- 2026-04-01: 今天天气不错
- 2026-04-01: 吃了拉面
- 2026-04-02: 会议记录...
（膨胀到 15,000 字符，每次对话浪费大量 Token）
```

**正确做法**：MEMORY 只存"黄金级信息"（长期有效的关键事实），日志存到 `memory/` 目录。

```
# MEMORY.md（精简，< 3000 字符）
## 用户偏好
- 主要语言：TypeScript
- 代码风格：函数式优先

## 项目信息
- 当前主项目：auth-service
- 部署方式：Docker + K8s
```

- 详细说明 → [[记忆系统]]

### 2. 模型选最大最强

> [!warning] 严重程度：中
> 所有场景都用 Opus，响应慢（30-60秒）且贵（$15/$75 per M tokens）。简单查询用 Opus 完全是浪费。

**正确做法**：按任务复杂度选择模型。

| 场景 | 推荐模型 | 响应时间 | 成本（估算，请以官方为准） |
|------|---------|---------|------|
| 闲聊、简单查询 | Haiku 4.5 | 1-3 秒 | 参见官方定价 |
| 日常对话、代码问答 | Sonnet 4.6 | 5-15 秒 | 参见官方定价 |
| 复杂推理、长文分析 | Opus 4.6 | 30-60 秒 | 参见官方定价 |

- 详细策略 → [[模型配置策略]]

### 3. DM 策略用 open

> [!danger] 严重程度：高
> `dmStrategy` 设为 `open`，任何人都能和你的 bot 对话。可能被滥用、被刷请求、泄露对话内容。

```json
// 错误示范：完全开放
{
  "dmStrategy": "open"
}
```

**正确做法**：使用 `pairing` 或 `allowFrom` 限制访问。

```json
// 正确做法：仅允许指定用户
{
  "dmStrategy": "allowFrom",
  "allowedUsers": ["+8613800138000", "user1@example.com"]
}
```

- 详细配置 → [[安全最佳实践]]

### 4. Docker 不设资源限制

> [!warning] 严重程度：中
> 容器没有配置 memory/cpus 限制，异常情况下可能吃光宿主机资源，导致其他服务不可用。

```bash
# 错误示范：无限制
docker run -d openclaw

# 正确做法：配置资源限制
docker run -d \
  --memory="2g" \
  --cpus="2" \
  --memory-swap="2.5g" \
  openclaw
```

- 详细部署 → [[Docker部署]]

### 5. 技能不设超时

> [!warning] 严重程度：中
> 浏览器自动化技能（如网页抓取、截图）卡住时没有超时机制，会阻塞整个会话，bot 无法处理其他消息。

```json
// 错误示范：无超时
{
  "name": "browse-web",
  "handler": "scripts/browse.js"
  // 没有超时配置，可能永远卡住
}
```

**正确做法**：所有技能设置超时，推荐 30-60 秒。

```json
{
  "name": "browse-web",
  "handler": "scripts/browse.js",
  "timeout": 30000
}
```

- 详细开发 → [[自定义技能开发]]

### 6. 不配 fallback 模型

> [!warning] 严重程度：中
> 主模型 API 故障时（Anthropic 偶尔出现），整个 bot 直接宕机无法响应。

```json
// 错误示范：单点故障
{
  "model": "claude-sonnet-4-6"
  // Anthropic API 挂了 → bot 宕机
}
```

**正确做法**：配置至少 1 个 fallback 模型。

```json
{
  "models": {
    "primary": "claude-sonnet-4-6",
    "fallbacks": [
      "gpt-4o",
      "claude-haiku-4-5"
    ]
  }
}
```

- 详细配置 → [[模型配置策略]]

---

### 7. 不配置日志轮转

> [!tip] 被忽视的运维问题
> OpenClaw 日志持续增长，几个月后日志文件可能达到数 GB，磁盘写满导致服务异常。

**正确做法**：配置 Docker 的 log driver 限制日志大小。

```bash
docker run -d \
  --log-driver json-file \
  --log-opt max-size="10m" \
  --log-opt max-file="3" \
  openclaw
```

### 8. workspace 路径配置错误

> [!warning] 严重程度：低
> Docker 容器内的 workspace 路径和宿主机挂载路径不匹配，导致记忆系统找不到文件。

```bash
# 错误示范：路径不一致
docker run -v /home/user/data:/app/data openclaw
# 容器内配置指向 /home/node/.openclaw，但数据在 /app/data

# 正确做法：确保挂载路径和配置路径一致
docker run -v ~/.openclaw:/home/node/.openclaw openclaw
```

## CC + OC 协同陷阱

### 1. API Key 没复用

> [!warning] 严重程度：中
> Claude Code 和 OpenClaw 各自购买独立的 API 额度，余额分散管理，容易造成浪费或某一侧余额不足。

**正确做法**：共享同一个 API Key，通过 `BASE_URL` 中转统一管理用量。

```bash
# Claude Code
export ANTHROPIC_API_KEY=sk-ant-xxx

# OpenClaw（复用同一个 Key）
# openclaw.json 中引用环境变量
{
  "models": {
    "apiKey": "${ANTHROPIC_API_KEY}"
  }
}
```

- 详细协同 → [[CC-OC协同工作流]]

### 2. 记忆体系割裂

> [!warning] 严重程度：中
> Claude Code 的记忆（CLAUDE.md、对话上下文）和 OpenClaw 的记忆（MEMORY.md、memory/ 目录）完全独立，关键信息无法互通。

**正确做法**：通过 MCP 桥接或 Webhook 同步关键信息。

```
方案 1：MCP 桥接
Claude Code → MCP Server → 读写 OpenClaw 的 MEMORY.md

方案 2：Webhook 同步
OpenClaw 技能 → 调用 Webhook → 更新 CC 的项目文档

方案 3：共享存储
两者都挂载同一个文件系统目录
```

- 记忆系统说明 → [[记忆系统]]
- 协同配置 → [[CC-OC协同工作流]]

### 3. 技能重复开发

> [!tip] 常见浪费
> Claude Code 写了一个代码审查 Skill，OpenClaw 又写了一个类似的审查 Skill，逻辑重复且维护成本翻倍。

**正确做法**：共享 Skill 逻辑，通过 MCP 复用。

```
 Claude Code Skill        OpenClaw Skill
        |                       |
        v                       v
  ┌─────────────────────────────┐
  │   共享逻辑层（独立脚本/包）    │
  │   scripts/review-core.js    │
  └─────────────────────────────┘
        |                       |
   CC 调用本地脚本          OC 通过 MCP 调用
```

- Skill 系统说明 → [[技能系统]]

### 4. 忽视版本同步

> [!warning] 严重程度：低
> Claude Code 更新到新版本，OpenClaw 还在用旧版本的 SDK，接口不兼容导致集成失败。

**正确做法**：建立版本检查机制，两个工具保持同步更新。

```bash
# 定期检查版本
claude --version
openclaw --version

# CI 中加入版本检查
# .github/workflows/check-versions.yml
```
