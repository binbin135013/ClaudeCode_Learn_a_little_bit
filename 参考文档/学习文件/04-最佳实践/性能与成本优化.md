# 性能与成本优化

> [!summary] 一句话定义
> 管理上下文窗口和 Token 消耗的策略，平衡**响应速度、回答质量和使用成本**。

> [!warning] 数据说明
> 本文中的 Token 数量、百分比和成本数据为估算值，基于社区经验和一般推理，非官方数据。实际消耗取决于具体使用场景。定价请以 [Anthropic 官方定价页](https://www.anthropic.com/pricing) 为准。

---

## 上下文窗口构成

```
系统提示词      ~2,000 tokens
CLAUDE.md      ~1,000-5,000 tokens
对话历史        动态增长 ⚠️
工具结果        动态增长 ⚠️
当前消息        用户输入
─────────────────────────────
总计上限：1,000,000 tokens (Sonnet 4.6)
```

## 四大优化策略

### 1. 精简 CLAUDE.md
- 目标 < 5000 字符 → [[CLAUDE-md编写指南]]
- 用列表代替段落
- 引用外部文件而非内联

### 2. 对话管理
```bash
/clear                    # 清理对话
claude --new-session      # 新会话
/tp                       # 跨设备传送
```

### 3. 任务分解
- 大任务拆成小步骤
- 每步完成后 `/clear` 再开始下一步
- 避免单次请求过大

### 4. 模型选择

| 任务 | 推荐模型 | 成本（估算，请以官方为准） |
|------|---------|------|
| 简单生成 | Haiku 4.5 | 参见官方定价 |
| 日常开发 | Sonnet 4.6 | 参见官方定价 |
| 复杂推理 | Opus 4.6 | 参见官方定价 |

## 成本计算

```
单次对话成本 = (输入tokens/1M × 输入价) + (输出tokens/1M × 输出价)

示例 (Sonnet 4.6)：
10K 输入 + 2K 输出
= (10,000/1M × $3) + (2,000/1M × $15)
= $0.03 + $0.03 = $0.06
```

## Prompt Caching

> [!note] API 层自动优化
> - 缓存写入：1.25x 价格
> - 缓存读取：0.1x 价格（节省 90%）
> - 对重复上下文（如 CLAUDE.md）特别有效

## 批量处理

```bash
# ❌ 逐个处理（多次调用）
claude "检查 a.ts"
claude "检查 b.ts"

# ✅ 批量处理（一次调用）
claude "检查 src/ 下所有 .ts 文件的代码质量"
```

> [!warning] 128K 输出注意
> Opus 支持最大 128K 输出 token，大规模生成时注意监控输出成本。

## 进阶：高级优化策略

### 上下文窗口深度管理

多轮对话中上下文窗口如何增长，以及对应的控制手法：

```
每轮对话增量:
  用户消息          ~200-500 tokens
  Claude 回复       ~300-1500 tokens
  工具调用结果       ~500-5000+ tokens  ← 大头

示例 — 10 轮对话后的上下文累积:
  系统提示词         ~2,000 tokens
  CLAUDE.md         ~3,000 tokens
  10 轮对话历史      ~15,000 tokens
  工具结果累积       ~30,000+ tokens
  ──────────────────────────────
  总计              ~50,000+ tokens
```

> [!tip] 核心原则
> 上下文越大，每次请求的输入成本越高、推理速度越慢。**主动管理上下文是省钱提速的关键**。

**优化手法**:

| 手法 | 节省效果 | 适用时机 |
|------|---------|---------|
| `/compact` | 显著压缩历史 | 长对话中途，感觉响应变慢时 |
| `/clear` + 新需求 | 100% 重置 | 切换任务、切换文件时 |
| 精准 Grep 代替 Read | 大幅减少工具输出 | 搜索代码、确认某行是否存在时 |
| `head_limit` 参数 | 限制搜索结果数 | Grep/Glob 查询，只需确认匹配存在时 |
| 精确描述需求 | 减少无效工具调用 | 每次提问前想清楚要什么 |

> [!warning] 常见反模式
> 在长对话中反复让 Claude "读一遍整个文件"——每次 Read 的结果都会累积在上下文中。改用 Grep 精准定位，或 `/compact` 后再继续。

### Prompt Caching 详解

> [!note] Prompt Caching 工作原理
> Anthropic API 自动缓存前缀相同的请求。对于 Claude Code 这意味着：
> - CLAUDE.md 内容每次请求都相同 → 几乎 100% 缓存命中
> - 系统提示词 → 缓存命中
> - 历史对话（不变化的部分）→ 缓存命中
>
> 缓存命中后，输入价格从 $3/M 降至 $0.3/M（Sonnet 4.6），即 **90% 折扣**。

**具体计算示例**:

```
无缓存费用（10K CLAUDE.md + 5K 历史 + 1K 新消息）:
输入: 16K × $3/M = $0.048

有缓存费用（假设 80% 命中）:
缓存输入: 12K × $0.3/M = $0.0036
普通输入: 4K × $3/M = $0.012
总计: $0.0156

节省: 可节省相当比例的成本（具体比例因对话长度而异）
```

> [!tip] 实际影响
> 缓存效果随对话变长越来越显著。对话前期（历史短）节省约 30-40%，对话后期（历史长）节省可达 70-80%。这也是为什么 `/clear` 后前几轮反而"更贵"——缓存还没建立起来。

### 模型路由策略

使用 [[Claude Code]] Agent Teams 时，不同子任务应路由到不同模型以优化成本和质量：

| 子任务类型 | 推荐模型 | 原因 |
|-----------|---------|------|
| 代码搜索/读取 | Haiku | 简单匹配，不需要推理 |
| 代码生成 | Sonnet | 平衡质量和速度 |
| 架构设计 | Opus | 需要深度推理和多步规划 |
| 代码审查 | Opus | 需要发现隐含问题和边界情况 |
| 格式化/重构 | Sonnet | 规则明确，执行为主 |
| 文档生成 | Haiku/Sonnet | 模板化程度高，不需要深度创意 |

> [!note] 在 OMC 中的实践
> oh-my-claudecode 的 Agent Teams 内置了模型路由：`explore` 用 Haiku，`executor` 用 Sonnet，`architect` 用 Opus。你也可以在 `settings.json` 中自定义每个 agent 的模型。详见 → [[模型配置策略]]

### OpenClaw 性能优化

针对 [[OpenClaw]] 部署的专项优化：

- **模型选择** — 简单对话用 Haiku，复杂任务用 Sonnet，仅在需要深度推理时启用 Opus。通过模型配置实现自动路由 → [[模型配置策略]]
- **记忆文件控制** — `MEMORY.md` 保持 < 2000 字符，日志只保留最近 7 天。过大的记忆文件会显著增加每次请求的 token 消耗 → [[记忆系统]]
- **技能精简** — 不用的技能不加载。每个技能的描述和示例都会占用 prompt 空间，精简技能列表可以减少每次请求的 token 数量
- **Gateway 超时** — 配置合理的超时阈值防止请求堆积。建议设置：简单任务 30s，中等任务 60s，复杂任务 120s → [[架构设计]]

> [!warning] OC 部署注意
> OpenClaw 的 Gateway 会缓存模型响应，合理利用缓存可以减少重复请求的延迟和成本。确保 Gateway 配置中启用了响应缓存。
