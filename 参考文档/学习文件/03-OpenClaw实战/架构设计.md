# OpenClaw 架构设计

> [!summary] 一句话定义
> OpenClaw 采用**三层架构**：消息平台层 → Gateway 网关层 → AI 模型层，核心是 Gateway 控制平面。

---

## 三层架构

```
┌─────────────────────────────────────────────┐
│          消息平台层 (15+ Channels)            │
│  WhatsApp · Telegram · Discord · Slack · 飞书  │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│          Gateway (ws://127.0.0.1:18789)       │
│  路由 · 会话 · 安全 · 技能 · 记忆 · 工具       │
│  Cron · Webhook · 媒体 · Control UI           │
└──────────────────┬──────────────────────────┘
                   │
          ┌────────▼────────┐
          │  Pi Agent Runtime │
          └────────┬────────┘
                   │
┌──────────────────▼──────────────────────────┐
│          AI 模型层 (29+ Providers)             │
│  Claude · GPT · Gemini · Ollama · DeepSeek     │
└─────────────────────────────────────────────┘
```

## Gateway 核心职责

| 模块 | 职责 | 关键配置 |
|------|------|---------|
| **连接管理** | 管理消息平台长连接 | Channel 配置 |
| **会话管理** | main 会话、群聊隔离 | Session 配置 |
| **Agent 调度** | 多 Agent 路由 | Agent 配置 |
| **安全控制** | DM 配对、允许列表 | allowFrom |
| **技能系统** | 加载和执行技能 | Skills 配置 |
| **记忆系统** | 加载和管理记忆 | Memory 插件 |

## 消息流转

```
1. 消息到达 Channel
2. Baileys/grammY 转换为统一格式
3. Gateway 路由判断（Agent归属 + 权限检查）
4. 会话管理器加载上下文 + 记忆 + 技能
5. Pi Agent 推理循环（Prompt → 模型 → 工具调用 → 循环）
6. 流式输出推送到 Channel
7. 状态持久化（会话历史 + 记忆文件）
```

> [!tip] 关键设计决策
> - **每个会话串行执行** — 防止竞态条件
> - **流式输出** — 实时显示 AI 回复
> - **工具调用循环** — AI 可连续调用多个工具

## 进阶：数据流与性能优化

### 一条消息的完整旅程

```
1. 用户在 Telegram 发送 "帮我查明天天气"
2. grammY 接收 → 转为统一消息格式
3. Gateway 路由引擎：
   ├── 匹配 Agent（default）
   ├── DM 配对检查（已批准？）
   └── 群聊/私聊判断
4. 会话管理器：
   ├── 加载历史上下文
   ├── 加载 MEMORY.md + 今日日志
   └── 加载 weather 技能指令
5. Pi Agent 推理循环：
   ├── 组装 Prompt（系统+记忆+历史+消息）
   ├── 调用 Claude API
   ├── Claude 返回工具调用请求 → weather 工具
   ├── 执行工具 → 获取天气数据
   └── 将结果喂回 Claude → 生成最终回复
6. 流式输出：实时推送到 Telegram
7. 持久化：
   ├── 会话历史写入磁盘
   ├── 如果 AI 决定记住 → 写入 memory/2026-04-02.md
   └── 使用统计更新
```

### 性能瓶颈与优化

| 瓶颈 | 症状 | 优化方案 |
|------|------|---------|
| 模型响应慢 | 回复延迟 > 5s | 切换更快模型(Haiku/Sonnet) |
| 上下文过大 | Token 消耗高 | 精简技能指令、减少历史加载 |
| 工具执行慢 | 浏览器操作超时 | 增加超时配置、使用缓存 |
| 记忆搜索慢 | 回复前卡顿 | 定期清理旧日志、精简 MEMORY.md |
| 并发瓶颈 | 消息排队处理 | 每个会话串行执行是设计决策，无法并行 |

> [!tip] 生产环境优化清单
> - [ ] 使用 Sonnet 替代 Opus（速度提升 2-3x）
> - [ ] 定期清理 `memory/` 目录（保留最近 7 天）
> - [ ] 精简 MEMORY.md（控制在 2000 字符内）
> - [ ] 配置模型故障转移（主模型挂了自动切备用）
> - [ ] 启用 Gateway Token 认证 → [[安全最佳实践]]
